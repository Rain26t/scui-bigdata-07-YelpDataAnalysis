02. B2B Data Analysis and Visualization
What Is Yelp?
Founded in 2004, Yelp is a popular online directory for discovering local businesses ranging from bars, restaurants, and cafes to hairdressers, spas, and
gas stations.

How Does Yelp Work?
You can search Yelp via its website or with the official apps on iOS and Android smart devices. Listings are sorted by business type and results are filtered
by geographical location, price range, and unique features like outdoor seating, delivery service, or the ability to accept reservations.
Yelp has a strong social aspect and encourages its users to leave written reviews, star ratings, and photos of their experience with each business they visit.
Each Yelp account has a friends list that can be populated by connecting the app with Facebook and a smartphone or tablet's address book. Reviews
posted on Yelp can also be reviewed by other users, while popular reviewers have the potential to be promoted to Yelp Elite status.

Yelp Dataset
The Yelp dataset is a subset of Yelp's businesses, reviews, and user data that has been made publicly available for use for personal, educational, and
academic purposes. Available as JSON files, use it to teach students about databases, to learn NLP, or for sample production data while you learn how to
make mobile apps. This dataset contains 6,685,900 reviews, 192,609 businesses, 200,000 pictures in 10 metropolitan areas.

Requirement 2 : Data Enrichment and Exploration
In the real-world tech industry, top-tier data analysts never limit themselves to a single internal database. They master the art of Data Enrichment—
seamlessly integrating internal business data with external real-world datasets (such as macroeconomics, weather, geography, or public records) to deliver
game-changing business insights.
You must independently design your SQL/Spark logic to process massive Yelp datasets and mandatorily incorporate an external dataset (or real-world
evidence) for cross-validation!

I.The Weather-Mood Hypothesis
Business Hypothesis: Do extreme weather conditions (heavy rain, strong winds, extreme heat) cause restaurants to receive a spike in 1-star
reviews on that specific day? Alternatively, do they trigger a surge in foot traffic/check-ins for specific food categories (like delivery or hot soup)?
Requirements: Analyze the true impact of historical weather fluctuations on local restaurant foot traffic (check-in activity) and reputation (star
ratings and review sentiment). You must prove through data integration whether diners' "moods" and "consumption preferences" are influenced
by Mother Nature. Finally, output actionable business recommendations for inventory management or staff scheduling based on forecasted
weather conditions.
Data Hunting:
Internal Data (Yelp): Extract time-stamped reviews, ratings, text, and daily check-in records for a specific city.
External Data (External): Independently navigate to NOAA (National Oceanic and Atmospheric Administration), Kaggle, or a local
meteorological website to download the Historical Daily Weather Data for that specific city and year.

Requirements: Pinpoint those frequently failing "cursed storefronts" alongside the low-turnover "golden locations" within the massive dataset.
You must conduct a comprehensive "post-mortem diagnosis" of these extreme addresses' life-and-death cycles by combining the merchants'
objective attributes, subjective diner reviews (NLP), and external spatial or economic dimensions.
Data Hunting:

Requirement 2 : Full-Stack Text-to-SQL Conversational Application
In this phase, you will step away from the Zeppelin notebook environment and build an independent, conversational AI web application tailored specifically
for the Yelp Dataset. This phase tests your software engineering skills, API integration, and Prompt Engineering by bringing your data to life through a
chat interface.
You have complete freedom to choose your frontend and backend frameworks (e.g., Streamlit for a pure Python approach, or a decoupled architecture
using Vue.js/React for the frontend and FastAPI/Node.js/Spring Boot for the backend). The application must feature a chat-based interface that integrates
a Large Language Model (LLM) API (e.g., OpenAI, Claude, or local models via Ollama) with your Yelp big data backend.

I. System Architecture & Technical Requirements
To successfully implement the Text-to-SQL pipeline for the Yelp data, your application must consist of the following logical layers:
Presentation Layer (Frontend): A chat-style user interface that captures user queries about restaurants and users, displays loading states, and
renders multimodal responses (text, generated SQL code, tabular results, and charts).
Service Layer (Backend): A server or application logic layer responsible for receiving frontend requests, constructing the LLM prompts with Yelp
context, handling API limits/keys, and communicating with your database.
Data & Execution Layer (Database): The underlying database engine (e.g., MySQL, SQLite, Hive, or Spark SQL) hosting the cleansed Yelp
tables (business, review, user, checkin, etc.) that executes the AI-generated SQL and returns the raw result sets.

II. The Text-to-SQL Workflow (Core Logic)
Your backend must programmatically orchestrate the following workflow:
Dynamic Schema Injection: Before sending the user's natural language question to the LLM, the backend must dynamically fetch or construct
the database schema specific to Yelp (e.g., clarifying that stars and review_count belong to the business table, or providing sample values
for categories like 'Mexican' or 'Nightlife') and inject it into the LLM's system prompt.
Natural Language Querying: Users can ask analytical questions about the Yelp ecosystem in plain English.
Example: "Show me the top 5 highest-rated Mexican restaurants in Philadelphia that have over 500 reviews."
Example: "Which 10 cities have the most elite users?"
SQL Generation & Sanitization: The LLM must translate the user's intent into a valid SQL query specific to your chosen database dialect. The
backend must parse the LLM's response to extract only the executable SQL code (removing markdown tags like ```sql or conversational filler).
Execution & Error Handling (Self-Correction): The backend must execute the generated SQL against the Yelp database. Crucial: If the SQL
execution fails (e.g., due to a syntax error or hallucinating a column like restaurant_name instead of name), your application should ideally
catch the error log, feed it back to the LLM, and ask it to correct the SQL automatically before returning a failure message to the user.

III. UI/UX & Data Presentation Deliverables
The frontend interface must go beyond simple text generation. It must provide a rich, interactive data-exploration experience for the Yelp dataset. The UI
must include:
Conversational Chat Layout: A standard chatbot interface with distinct user bubbles and AI response bubbles.
Loading Indicators: Clear visual feedback (e.g., a spinner or "Analyzing Yelp reviews...") to inform the user that complex SQL execution (like
joining the user and review tables) is taking place in the background.
Transparent Code Display: The generated SQL code must be displayed to the user within a formatted, syntax-highlighted code block so the
user can verify how the business or user metrics were calculated.
Tabular Data Rendering: The raw results returned from the database (e.g., a list of businesses with their addresses and average ratings) must
be formatted and rendered as an interactive, scrollable HTML or component-based data table.

Automated Chart Generation (Bonus/Advanced): If the returned data is suitable for visualization (e.g., counting businesses by city, or
averaging stars by categories), the frontend should automatically map the data to a charting library (like ECharts, Chart.js, or Streamlit native
charts) to display a bar, line, or pie chart alongside the data table.
